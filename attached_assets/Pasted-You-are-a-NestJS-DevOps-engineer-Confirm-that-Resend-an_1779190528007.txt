CONTEXT:
PROJECT_STATUS.md flags Resend and Twilio as "needs production credentials."
If these are not set, the vendor approval email — the email that gives a
vendor access to the portal — silently fails. This is the single highest-risk
gap before onboarding any vendor.

STEP 1 — Confirm Resend credentials in Replit Secrets
In the Replit deployment secrets (not workspace secrets), confirm:
  RESEND_API_KEY = re_... (production key, not test key)
  RESEND_FROM_EMAIL = hello@feastpot.co.uk (or noreply@feastpot.co.uk)
  RESEND_FROM_NAME = Feastpot

If RESEND_API_KEY starts with re_test_ — it is a test key.
Replace it with the live production key from resend.com/api-keys.

STEP 2 — Confirm Resend domain is verified
Log in to resend.com → Domains.
feastpot.co.uk must show status: Verified (green).
If it shows Pending or Unverified:
  Add the required DNS TXT and MX records to your DNS provider.
  Without domain verification, Resend will reject all outbound emails
  from @feastpot.co.uk addresses.

STEP 3 — Fire a live test of the vendor portal invite template
Add a temporary test endpoint to the API:

File: apps/api/src/modules/admin/admin.controller.ts

@Post('admin/test/vendor-invite-email')
@Roles('admin')
async testVendorInviteEmail(@Body() dto: { email: string; name: string }) {
  if (process.env.NODE_ENV === 'production') {
    // Allow in production but require admin role — this is ops tooling
  }
  const { subject, html } = vendorApprovedTemplate({
    vendorFirstName: dto.name,
    businessName:    'Test Kitchen',
    portalUrl:       `${process.env.VENDOR_PORTAL_URL}/onboarding`,
    supportEmail:    'support@feastpot.co.uk',
  })
  await this.notificationsService.sendEmail({
    to:      dto.email,
    subject,
    html,
  })
  return { sent: true, to: dto.email, subject }
}

Run: curl -X POST https://feastpot-platform.replit.app/v1/admin/test/vendor-invite-email \
  -H "Authorization: Bearer {admin_token}" \
  -H "Content-Type: application/json" \
  -d '{"email":"soul@feastpot.co.uk","name":"Soul"}'
