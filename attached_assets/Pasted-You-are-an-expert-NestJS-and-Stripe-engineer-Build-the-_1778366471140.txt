1. orders.service.ts with these key methods:

createOrder(customerId, dto: CreateOrderDto):
- Validate all menuItemIds exist and are isAvailable=true
- Validate delivery slot: scheduledFor must be within vendor's slot_open_time/slot_close_time on an available day, at least leadTimeHours from now
- Calculate: subtotalPence = sum(quantity * unitPricePence), deliveryFeePence from DeliveryConfig (free if above freeAbovePence threshold), discountPence (from discount code if provided), totalPence
- Calculate: commissionPence = Math.round(totalPence * vendor.commissionRatePct / 100), netToVendorPence = totalPence - commissionPence
- Call stripe.paymentIntents.create({ amount: totalPence, currency: 'gbp', capture_method: 'manual', metadata: { orderId } })
- Save order with stripePaymentIntentId, status: pending
- Return { order, clientSecret: paymentIntent.client_secret }

confirmOrder(orderId, customerId):
- Verify order belongs to customer and status=pending
- Update status to pending (awaiting vendor)
- Enqueue notification job: { type: 'notify_vendor', vendorId, orderId }
- Enqueue BullMQ delayed job: { type: 'auto_cancel', orderId, delay: 15 * 60 * 1000 }

updateStatus(orderId, status, userId):
STATUS MACHINE — only these transitions are valid:
vendor (own vendor's orders): pending → accepted, pending → rejected, accepted → preparing, preparing → dispatched, dispatched → delivered
admin: any → cancelled, any → refunded

On accepted: cancel the auto_cancel delayed job
On delivered: call stripe.paymentIntents.capture(stripePaymentIntentId), update commissionPence and netToVendorPence, enqueue review_trigger job with 2-hour delay

On rejected: call stripe.paymentIntents.cancel(), update cancellationReason

reorder(originalOrderId, customerId, overrides?):
- Fetch original order with items
- Validate all items still exist and are available
- Call createOrder with same items and vendor, new scheduledFor
