You are a Next.js 15 engineer. Fix a Vercel production deployment failure.

CONFIRMED ERROR FROM VERCEL BUILD LOGS:

  npm error workspace @feastpot/web@@0.1.0

ROOT CAUSES (two, both must be fixed):
1. The homepage makes API calls at build time (to fetch vendors, featured kitchens etc.)
   that either time out or fail in the Vercel build environment where NEXT_PUBLIC_API_URL
   points to the production API — which may not be reachable during the build step.

2. Something in the homepage route (or a component it imports) declares
   `export const runtime = 'edge'` or uses a Next.js 15 API that forces edge runtime,
   which conflicts with static generation.

════════════════════════════════════════════════════════
STEP 1 — Force the homepage to be dynamic (not statically generated)
File: apps/web/src/app/page.tsx
════════════════════════════════════════════════════════

Add these two exports at the TOP of apps/web/src/app/page.tsx,
before any imports or the component definition:

export const dynamic = 'force-dynamic'
export const revalidate = 0

This tells Next.js: do NOT attempt to pre-render this page at build time.
Render it at request time instead (SSR). This immediately stops the 60-second timeout.

The homepage fetches vendor data from the API — that data changes constantly
(new vendors, ratings, availability). It should never be statically generated anyway.

════════════════════════════════════════════════════════
STEP 2 — Find and fix any edge runtime declarations
════════════════════════════════════════════════════════

Run this in the Replit shell:
grep -rn "runtime = 'edge'\|runtime = \"edge\"" apps/web/src --include="*.ts" --include="*.tsx"

For EVERY file that contains `export const runtime = 'edge'`:

If it is apps/web/src/app/icon.tsx or apps/web/src/app/apple-icon.tsx:
  These are fine — image response routes SHOULD use edge runtime.
  Leave them alone.

If it is apps/web/src/app/page.tsx or any page component:
  Remove the runtime = 'edge' export entirely.
  The force-dynamic export from Step 1 is sufficient.

If it is apps/web/src/app/auth/callback/route.ts:
  This is fine — route handlers can use edge runtime.
  Leave it alone.

════════════════════════════════════════════════════════
STEP 3 — Make all data-fetching on the homepage client-side
File: apps/web/src/app/page.tsx
════════════════════════════════════════════════════════

The homepage must NOT fetch data in the server component body during build.
All API calls must be in 'use client' components using TanStack Query (useQuery),
so they run in the browser at runtime, not at build time.

Check the current page.tsx:

IF the homepage is a Server Component (no 'use client') that calls fetch() directly:

Convert it to use the existing client-side hooks:

'use client'
import { useQuery } from '@tanstack/react-query'

// Replace any direct fetch() calls with useQuery hooks:
const { data: featuredVendors, isLoading: vendorsLoading } = useQuery({
  queryKey: ['homepage-vendors'],
  queryFn: () => fetch(`${process.env.NEXT_PUBLIC_API_URL}/v1/vendors?limit=6&communityFavourite=true`)
    .then(r => r.json()),
  staleTime: 5 * 60 * 1000,  // 5 minutes
})

const { data: newVendors } = useQuery({
  queryKey: ['homepage-new-vendors'],
  queryFn: () => fetch(`${process.env.NEXT_PUBLIC_API_URL}/v1/vendors?limit=6&sortBy=newest`)
    .then(r => r.json()),
  staleTime: 5 * 60 * 1000,
})

IF the homepage is already 'use client':
  The issue is that a child Server Component is making the slow fetch.
  Add `export const dynamic = 'force-dynamic'` to page.tsx and that is sufficient.
